When evaluating an application for suitability for the cloud, one must consider the existing investment. There is likely to be significant costs associated with servers and support infrastructure. The migration of a server's functionality to the cloud may leave it defunct and a wasted business investment unless the business can somehow repurpose the servers.

Servers are likely to be on a hardware lifecycle roadmap whereby they are refreshed every 3-5 years, so that may be the ideal time to migrate on-premises resources to the cloud.

2. Consider application performance and portability

In the case of application servers, we must consider whether the application can function in the cloud. Compatibility and application performance are not likely to be a problem in the GCP environment, but you must consider two important aspects: latency or delay and applications portability.

The first aspect is latency or delay because even though you can provision the virtual instance for the application server with nearly unlimited compute and memory resources if there is limited internet bandwidth then that may well be detrimental to the application performance. The second aspect to consider will be the applications portability. In this case, migrate a virtualized application server to the cloud will not be a problem but the application might have some important external dependencies.

3. Assess the network premises and how it will integrate with the cloud

If an organization wants their users to be able to access applications in premises and some in the cloud, the cloud network must function seamlessly or if they wish to keep some of their resources on as an extension of the on-premises network and vice versa. Typically, you have to deploy cloud-based AD domain controllers, DNS, and maintain a secure communications path between the cloud and the on-premises network.

It is essential to understand that when building a migration plan, you do not usually need to move everything at the same time. It's crucial to strive for a lift and shift of easy applications and like-for-like databases early on, which is the low-hanging fruit. Then, as you build experience and confidence, you can start using Compute Engine VMs to migrate virtualization of other legacy applications or databases that aren't directly supported in GCP.

For example, MySQL and PostgreSQL can be considered straightforward since GCP supports these databases natively. This is what would be described as a homogenous migration. However, if you need to later move a Microsoft SQL Server cluster or an Oracle server to the cloud, you could run it within a VM on Compute Engine or a container in Kubernetes.

The point being, you don't need to lift and shift everything at once; take your time, and look for the best deployment methods on an application-to-application basis.

Evaluating Application Suitability for Cloud Migration


When considering cloud migration, it's essential to evaluate existing investments, including server costs and support infrastructure. Servers typically follow a hardware lifecycle roadmap, refreshing every 3-5 years, making this a suitable time for on-premises resources to be migrated to the cloud.


Application Performance and Portability


For application servers, consider compatibility and performance in the GCP environment. Two key aspects to evaluate are:


    

  • Latency: Even with unlimited compute and memory resources, limited internet bandwidth can negatively impact application performance.
    • 

  • Portability: Migrating virtualized application servers is not a problem, but external dependencies may require consideration. For example, MySQL databases with partner data source connections that rely on direct connections.
    • 



Assessing the Network


When building a cloud migration plan, consider integrating the premises network with the cloud:


    

  • Ensure users can access applications in both premises and the cloud.
    • 

  • Deploy cloud-based AD domain controllers and DNS.
    • 

  • Maintain secure communications between the cloud and on-premises networks.
    • 

  • Understand that not all resources need to be migrated at once; prioritize easy applications and databases (low-hanging fruit) first.
    • 



Homogenous vs. Heterogeneous Migration


Migration can be categorized into two types:


    

  • Homogenous migration: Migrating applications with similar requirements, such as MySQL or PostgreSQL, which are directly supported in GCP.
    • 

  • Heterogeneous migration: Migrating applications with different requirements, such as Microsoft SQL Server or Oracle servers, which may require running within a VM on Compute Engine or a container in Kubernetes.
    • 



Best Practices


    

  • Take time to evaluate the best deployment methods for each application.
    • 

  • Prioritize easy applications and databases (low-hanging fruit) first.
    • 

  • Build experience and confidence before migrating more complex applications.
    •
