Auditing existing architecture to identify security vulnerabilities is a crucial step in ensuring the overall security of an application. To start this process, one can use Cloud Security Scanner to surface known vulnerabilities and then work through those identified to remediate any issues.


However, some security vulnerabilities may be due to design flaws, which need to be addressed as well. Although Google Cloud Platform (GCP) has several important vulnerability controls in place, such as built-in security within the network, HTTP Global Load Balancing, and default VPC configurations, there are still gaps that need to be filled.


The first pass of the security audit identified an issue with the front-end security, which requires lockdown of external-facing webservers. Fortunately, some security techniques are already deployed, such as the default firewall and its rules when creating webserver VMs, as well as built-in network protections provided by Global Load Balancers.


To bolster DoS security further, adding Cloud CDN and Cloud DNS to the design can be beneficial. However, implementing auto-scaling for the front-end cluster is also essential to absorb attacks by adding additional servers.


The major risks identified include gaining unauthorized access to sensitive user information and hackers gaining access to potentially already having a Virtual Private Cloud with private Psubnet range. To prevent hackers from gaining a foothold on public-facing webservers, isolating them using private IP addressing can be an effective strategy.


By giving web servers private IPs, they will need a secondary NIC in the same private network as back-end servers, making it more difficult for hackers to access the back end. This approach can also be applied to lockdown backend storage by accessing GCS and CBT over the private network.


The security checklist has been updated with added firewalls and firewall rules to restrict access to certain ports, addressing the denial of service risk by adding auto-scaling to the front-end group. Private addresses have been given to backend servers, making them inaccessible from the internet, and the design is now in separate zones, allowing for isolation against fault tolerances.


However, there is also assumed implicit trust between all components on the network, which should be considered when utilizing security accounts via IAM.

Topic 1: Security Auditing and Remediation


This topic revolves around the process of identifying and addressing security vulnerabilities in an existing architecture. It involves using tools like Cloud Security Scanner to surface known vulnerabilities, working through identified issues to remediate them, and considering design flaws that may require additional attention.


Topic 2: Leveraging GCP Vulnerability Controls


This topic focuses on the built-in security features within Google Cloud Platform (GCP), such as HTTP Global Load Balancing, default VPC configurations, and firewall rules. It highlights how these controls can be leveraged to enhance security without requiring extensive additional configuration.


Topic 3: Addressing Front-end Security Risks


This topic specifically addresses the need to lock down external-facing web servers to prevent unauthorized access or use as a launching pad for attacks on internal application servers. It proposes using private IP addressing and secondary NICs in private networks to isolate these servers.


Topic 4: Implementing Private Network Segmentation


This topic emphasizes the importance of isolating backend storage, GCS, and CBT by accessing them over private networks. This ensures that sensitive data is protected from external threats and reduces the attack surface.


Topic 5: Enhancing Denial-of-Service (DoS) Security


This topic discusses ways to bolster DoS security further, including adding Cloud CDN and Cloud DNS to the design. It also highlights the need for implementing auto-scaling for front-end clusters to absorb attacks by adding additional servers.


Topic 6: Addressing Unauthorized Access Risks


This topic identifies gaining unauthorized access to sensitive user information as a major risk. It proposes utilizing security accounts via IAM to address this concern and ensure that components on the network are trusted only when necessary.


Topic 7: Implementing Auto Scaling for Front-end Clusters


This topic specifically addresses the need to implement auto-scaling for front-end clusters to absorb DoS attacks by adding additional servers. This ensures that the cluster can dynamically scale up or down in response to changing traffic demands.


Topic 8: Utilizing IAM Security Accounts


This topic emphasizes the importance of utilizing security accounts via IAM to address implicit trust between components on the network. It highlights the need for secure access controls and authentication mechanisms to ensure that only authorized users and services have access to sensitive resources.
