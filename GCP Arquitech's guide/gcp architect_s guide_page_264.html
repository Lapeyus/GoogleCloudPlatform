Workflow Migration

When first planning workflow migration you need to prioritize the machines you want to move. In order to do this with confidence, you will need to understand your application's dependencies. Then, you need to batch these dependencies within the same migration stage. For example: If an application depends on multiple VMs, migrate all of those VMs at the same time. However if that is not feasible then move the components logically. For example; if an application requires a database and a web server, then you can move the database before the web server.

In order to verify your migration strategy and tactics you could clone an on-premises workload and run it in Google Cloud Platform (GCP). This enables you to test the migration process without disrupting production. Nonetheless, the points you must consider when migrating workflows are:

• Move the VM to GCP and once this is done perform validation testing and resolve any issues.
• Migrate the application storage to GCP

Containerization Refactoring vs Serverless Application Refactoring


When migrating applications to the cloud, refactoring is often necessary to reap its benefits. Two modern techniques are containerization refactoring and serverless application refactoring.


Containerization Refactoring


This method involves using a technique called containerization refactoring, where applications are moved into containers with minimal modification. The concept behind this is that applications execute within containers, which also enable developers to insert cloud-native features and improve portability. Costs and refactoring times decrease due to the wide acceptance of containers in IT.


However, serverless platforms require code modifications to make applications work, as they don't yet support all languages and databases.


Serverless Application Refactoring


This trend has similar issues to containerization, such as learning new tools and technologies. However, with serverless refactoring, some code modifications are required to make the application work, due to limitations in supported languages and databases.


Technical Issues

Workflow Migration


When planning workflow migration, prioritize machines to move based on dependencies. Batch these dependencies within the same migration stage. For example, migrate all VMs at once if possible, or move components logically (e.g., database before web server).


To verify the migration strategy, clone an on-premises workload and run it in Google Cloud Platform (GCP) without disrupting production.
