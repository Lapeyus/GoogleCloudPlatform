Introduction


Service accounts are a way to authenticate and authorize access to Google Cloud resources. They can be used to grant permissions to users or groups.


Authentication Keys


There are two types of service account keys:


GCP-managed keys


    

  • Used by Google Cloud platform services, such as App Engine and Compute Engine.
    • 

  • Managed by Google Cloud.
    • 



User-managed keys


    

  • Created and managed by the user.
    • 

  • Can be used with default or custom service accounts.
    • 



Access Scopes vs. IAM Roles


Access scopes are a legacy method for specifying permissions on instances. They can only be customized for default service accounts, but not for user-created service accounts.


IAM roles, on the other hand, provide more granular permission control and can be assigned to groups or users.


Default Service Accounts


Default service accounts support primitive, project, and curated Cloud IAM roles.


Instance Admin Role


    

  • Has permissions to create, modify, and delete virtual machine instances, and disks.

  • Can be used as a starting point for customizing access scopes.

Assigning IAM Policies


To assign IAM policies to a service account:


    

  1. Create a service account with the desired IAM role.

  2. Assign the service account user role to users or groups.

Best Practices


    

  • Be cautious when granting the service account user role to users or groups.

Service accounts provide a secure way to authenticate and authorize access to Google Cloud resources. By understanding access scopes, IAM roles, and authentication keys, you can effectively manage permissions and ensure the security of your cloud-based applications.

Application A receives an access token with 'read-only' scope, so it can only read from the Cloud Storage bucket. Application B on the other hand, receives an access token with 'read-write' scope, so it can read and modify data in the Cloud Storage bucket.

These tokens can be short-term privileges so if you want to provide read access for only a limited time, say 24 hours, then you can create a 24 hour access token that your application service would use. Scopes can be further customized when you create an instance using the default service account. These scopes can also be changed after an instance is created by stopping it.

However, access scopes cannot be customized for user-created service accounts, but you can always use the granular Cloud IAM roles instead. This restriction is due to access scopes being the legacy method for specifying permissions on your instance. As before the development of the more advances IAM roles, access scopes were the only available mechanism for granting permissions to the service accounts.

Moreover, another distinction between service accounts is that default service accounts will support primitive, project, and curated, or Cloud IAM roles. While user-created service accounts only use Cloud IAM roles for access. Nonetheless, the roles for service accounts can also be assigned to groups or users as a service account can be looked upon as being a resource.

In this example, we will use a service account with the instance admin role, which has the permissions to create, modify, and delete virtual machine instances, and disks.

Service Accounts in Google Cloud Platform


Service accounts are a way to authenticate and authorize access to Google Cloud Platform (GCP) resources. They can be used by applications or services that need to interact with GCP without requiring human intervention.


Access Tokens and Scopes


When an application wants to use a GCP resource, such as a Cloud Storage bucket, it requests an access token from the Google authorization server. The access token is then used to authenticate the request. There are two types of access tokens: short-term privileges and long-term credentials.


    

  • Short-term privileges can be created for limited time periods, such as 24 hours.
    • 

  • Long-term credentials can be used for extended periods.
    • 



The scope of the access token determines what actions can be performed on the GCP resource. For example, an application with a 'read-only' scope can only read from the Cloud Storage bucket, while an application with a 'read-write' scope can both read and modify data in the bucket.


Customizing Scopes


Scopes can be customized when creating a service account instance using the default service account. However, access scopes cannot be customized for user-created service accounts. Instead, granular Cloud IAM roles can be used to specify permissions.


Assigning Roles to Service Accounts


Service accounts can be assigned to groups or users, allowing them to act on behalf of the service account. This is done by assigning the service account user role to a group or user.


    

  • Default service accounts support primitive, project, and curated Cloud IAM roles.
    • 

  • User-created service accounts only use Cloud IAM roles for access.
    • 



Authentication Keys


Service accounts use authentication keys to authenticate with GCP. There are two types of authentication keys:


    

  • GCP-managed keys: used by Google Cloud platform services such as App Engine and Compute Engine.
    • 

  • User-managed keys: can be created and managed by users.
    • 



In summary, service accounts provide a way for applications or services to authenticate and authorize access to GCP resources without requiring human intervention. By customizing scopes, assigning roles, and using authentication keys, service accounts can be used to manage access to GCP resources effectively.
