Cloud IAM Custom Roles and Service Accounts


Introduction


Cloud IAM provides custom roles that allow you to fine-tune permissions for specific users or groups. This section will cover the benefits, caveats, and usage guidelines for custom roles.


Benefits


    

  • Fine-grained control over permissions
    • 

  • Suitable for organizations adhering to a least privileged access model
    • 

  • Can be used at project or organization levels
    • 



Custom Roles


Custom roles are created to address specific requirements that may not be met by predefined roles. To create a custom role, you can add or remove permissions from an existing role.


Example: Instance Operator Role


InstanceOperator:
  * compute.instances.get
  * compute.instances.list
  * compute.instances.start
  * compute.instances.stop
  * compute.instances.setMachineType
  * compute.instances.setDisks
  * compute.instances.setNetworkInterfaces


Caveats


    

  • Custom roles require manual permission management, which can be complex and unpredictable.
    • 

  • Interaction between permissions can lead to error-prone and security-risky configurations.
    • 



Auditing Custom Roles


Some companies require custom roles to be thoroughly audited by a third party or peer review before assignment. Others prefer to stick with predefined roles due to the risks associated with custom roles.


Service Accounts


Service accounts are used to grant permissions to resources, rather than users. This is useful for automating processes that interact with multiple resources.


Example: Granting Permissions to a Compute Engine Virtual Machine


    

  • Create a service account with the necessary permissions
    • 

  • Assign the service account to the virtual machine
    • 

  • Use the service account to automate tasks and interact with the virtual machine
    • 



Usage Guidelines


    

  • Custom roles can only be used at project or organization levels.
    • 

  • Service accounts are suitable for automating processes that interact with multiple resources.
    •

By applying refined roles for example those provided for the Compute Engines VMs such as the InstanceAdmin Role, you can pass the required permissions to anyone designated that role to perform a certain set of actions on virtual machines. Thus they will be able to list all VMs within that project, read and modify their configurations, as well as change their state by starting and stopping them.

Custom Roles

However, if you designate a user to be in the InstantAdmin role for VMs that will apply to all instances of VMs within that project, it may still be too broad. To address this requirement, Cloud IAM introduces custom roles. The requirement for custom roles is driven by many companies adhering to a least privileged access model in which each person in the organization has the minimum amount of privilege needed to do their job.

So, for example, maybe you will need to define a Role that has the inherent permissions to allow a user to start and stop Compute Engine and virtual machines, but not be allowed to reconfigure them. This would be a suitable case for an Instance Operator role. Because of this requirement, Cloud IAM allows us to make custom roles, which allow you to customize a similar role by adding or removing some permission.

There is however a couple of caveats we need to be aware of when using custom roles. First, if you decide to use custom roles, you will need to manage their permissions manually. This can make custom roles error prone and even security risks. As a result, some companies require any custom roles to be thoroughly audited by a third party or a peer review before being assigned.

Another thing to be aware of is that custom roles can only be used at the project or organization levels, which rules them out at the folder level.

Service Accounts

So far in our discussion regarding Cloud IAM, we have been addressing the situation where a user or a group of users require identification, authentication and then given access to a resource to perform tasks compatible with their role. However, there is also the very common scenario where one resource needs to interact and communicate with another resource.

For example, you may need to give permissions to a Compute Engine virtual machine, rather than to a person, in order to automate a process. The way this is done is by using a service account.

Custom Roles and Service Accounts in Cloud IAM


Cloud IAM introduces custom roles to address the requirement for highly specific and fine-grained roles. This is driven by companies adhering to a least privileged access model, where each person has the minimum amount of privilege needed to do their job.


Custom Roles


Custom roles allow you to customize a similar role by adding or removing permissions. For example, you can create an Instance Operator role that has the inherent permissions to start and stop Compute Engine and virtual machines, but not reconfigure them.


However, custom roles have some caveats:


    

  • You need to manage their permissions manually, which can be complex and unpredictable.
    • 

  • Custom roles can only be used at the project or organization levels, ruling them out at the folder level.
    • 

  • Some companies require custom roles to be thoroughly audited by a third party or peer review before being assigned.
    • 



Service Accounts


Service accounts are used when one resource needs to interact and communicate with another resource. For example, you may need to give permissions to a Compute Engine virtual machine rather than to a person to automate a process.


By using service accounts, you can pass the required permissions to anyone designated that role to perform certain actions on virtual machines, such as listing all VMs within a project, reading and modifying their configurations, and changing their state by starting and stopping them.

Comprehension Questions


    

  1. 

    What is the purpose of custom roles in Cloud IAM?
Answer: Custom roles allow you to create fine-grained roles that can be tailored to specific needs, providing more control over access and permissions.

  2. 

    Why do companies adhere to a least privileged access model when using Cloud IAM?
Answer: Companies follow this model to ensure each person has the minimum amount of privilege needed to perform their job, reducing security risks and errors.

  3. 

    What is the main difference between custom roles and predefined roles in Cloud IAM?
Answer: Custom roles can be tailored to specific needs by adding or removing permissions, while predefined roles are pre-configured with a set of standard permissions.

Application Questions


    

  1. 

    Suppose you need to create a custom role that allows users to start and stop Compute Engine instances but not reconfigure them. What steps would you take?
Answer: You would define the custom role by adding or removing permissions from an existing role, such as InstanceAdmin or InstanceOperator, to achieve the desired level of access.
    

    2. 

  2. 

    How might companies use service accounts to automate processes in Cloud IAM?

Answer: Companies should thoroughly audit or peer-review custom roles before assignment, manage permissions manually, and consider the limitations of custom roles (e.g., only available at project or organization levels).
