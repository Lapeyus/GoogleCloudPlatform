Metrics and Scaling


Understanding Custom Metrics


The purpose of custom metrics is to get a Deployment to scale its replicated Pods based on the given metric value. However, metrics with a total target value should always be defined only when the effects of the scaling will bring the updated value of the metric closer to the target value.


Scaling a Frontend Application


We can see this better by considering how we might go about scaling a frontend application. We could do this by choosing a custom metric based on the queries-per-second metric. Thus, we are expecting that when the metric value increases, the number of Pods should scale up, with each Pod serving a similar amount of traffic as before.


Importance of Average vs Total Value


However, if we had exported from each pod the total number of queries-per-second and then set instead of an average but a total target value for this metric, it will not work as we hoped. This is because increasing the number of Pods doesn't have any effect on the total traffic value so the HPA will not scale the deployment based upon that metric.

Metrics used in practice can have unexpected results, so it's essential to understand how they work. Custom metrics are not always intuitive and should be defined only when scaling will bring the updated value closer to a target value.


For example, consider scaling a frontend application based on queries-per-second (qps) metric. By exporting qps values from each pod and setting a desired average target value, the deployment can scale up with each pod serving similar traffic. However, if the total number of qps is used instead of an average, increasing the number of pods will not affect the total traffic value, making it ineffective for scaling.


To address this issue, use metrics that are relevant to the specific use case and consider the effects of scaling on the metric's target value. In the context of a High Performance Computing analytics project, a collaborative application shared across multiple universities can benefit from a scalable architecture based on Kubernetes Engine clusters with auto-scale and pre-emptive VMs.


Key points:


    

  • Custom metrics should be defined to bring the updated value closer to a target value when scaling.
    • 

  • Metrics like queries-per-second (qps) require careful consideration of their relevance and how they affect scaling.
    • 

  • A highly scalable architecture based on Kubernetes Engine clusters with auto-scale and pre-emptive VMs can address scalability challenges in High Performance Computing analytics projects.
