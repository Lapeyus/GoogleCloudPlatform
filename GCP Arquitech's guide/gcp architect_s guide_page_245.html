Sharing and Isolation

As we demonstrated earlier, the least secure and least resilient design is where everything is hosted within a single failure domain. This is because all components in the design must directly interact, collaborate, and communicate to fulfill the design goals, making them fully dependent on one another.

There are many methods or techniques we can use to separate those inter-related parts by decoupling their dependencies on one another, producing more private or fault-tolerant communication channels between them. One method is to redesign using multiple failure domains, which inherently provides better isolation and resiliency. However, distribution of resources leads to increased latency, so we must also be aware of the need to share and collaborate on resources.

Thankfully, Google Cloud Platform provides many network topology features that can be used in designs to provide different blends of resource sharing and isolation. In the next section, we will take a deeper dive into resource sharing and isolation.

Defining Terms

For cloud architects, sharing is defined as an enabler for collaboration between components of the architecture. On the other hand, architects define isolation as a control that prevents the compromise of one part from spreading to the other. One very effective way to provide isolation is using a microservice architecture.

Microservices do not really communicate with other services as they have no concept of downstream or upstream services; they simply accept an input and hand off an output. Therefore, there is no effective communication with which to compromise another service. For example, should an exploiter gain access to one machine, they can't get into the others.

Natural Separation Techniques

One way to achieve this type of isolation is through leveraging the cloud. For instance, if you have two different projects under different administrative control but need to talk to each other, using a VPC means that they will have their own private IP address space. This allows you to isolate the two networks from each other's communication over the network, without authentication space and their own IAM.

Isolation Using VPN Tunnelling

In this scenario, we can have an encrypted link between two or more projects. While not a feature of Google Cloud Platform, if you want projects to talk to each other privately, it is used regularly in on-premises networks. You can use another private IP address subnet space to increase the privacy and even encrypt communications.

Cross-Project Virtual Private Cloud Network Peering

Then there's a new feature called cross-project Virtual Private Cloud network peering or as it was known, cross-project networking. In this case, network peering allows us to directly connect two projects together using a private IP address. This enables two different companies, two different organizations, to share and collaborate on resources and in projects.

Identity Access Controls

Hence, if you need to collaborate with an external partner and want to be able to communicate, you can set identity access controls. This gives you the ability to control what kind of communication can occur between these two particular domains.

Shared Private Cloud

And then there's the shared private cloud, which provides a secure environment for collaboration and data sharing among multiple organizations.

Sharing and Isolation in Cloud Networking


The least secure design is one where everything is hosted within a single failure domain, as all components must interact and collaborate to fulfill design goals, making them fully dependent on each other. To address this, we can use techniques such as decoupling dependencies, redesigning with multiple failure domains, and leveraging cloud networking features.


Defining Sharing and Isolation


In the context of cloud networking, sharing refers to collaboration between components of an architecture, while isolation refers to a control that prevents compromise from spreading to other parts. One effective way to provide isolation is through microservice architecture, where services do not communicate with each other due to their lack of concept of downstream or upstream services.


Achieving Isolation


Isolation can be achieved through various methods:


    

  • Leveraging the Cloud: Using Virtual Private Clouds (VPCs) provides private IP addresses for networks, isolating them from communication.
    • 

  • VPN Tunnelling: Encrypting links between projects using VPN tunnelling provides a secure connection.
    • 

  • Cross-Project Network Peering: Directly connecting two projects using a private IP address enables collaboration while maintaining isolation.
    • 



Shared Private Cloud


A shared private cloud allows multiple organizations to share resources and collaborate on projects, providing a secure and isolated environment for communication between domains. Identity access controls can be set to control what kind of communication occurs between these domains.


By understanding the importance of sharing and isolation in cloud networking, architects can design more resilient and secure architectures that balance collaboration with security.

Comprehension Questions


1. What is the primary reason why a single failure domain in design leads to security and resilience issues?


Answer: All components must directly interact, collaborate, and communicate to fulfill design goals, making them fully dependent on one another.


2. How does decoupling dependencies between components help improve communication channels?


Answer: Decoupling dependencies produces more private or fault-tolerant communication channels between components.


3. What is the main benefit of using a microservice architecture for isolation?


Answer: Microservices do not have effective communication with other services, making it difficult for an exploiter to gain access to one service and compromise others.


Analytical Questions


1. Compare and contrast the benefits of using multiple failure domains versus decoupling dependencies in design.


Answer: Multiple failure domains provide better isolation and resiliency but may lead to increased latency due to resource distribution, while decoupling dependencies improves communication channels without compromising security.


2. How does leveraging the cloud contribute to achieving isolation?


Answer: Cloud platforms offer various network topology features that enable different blends of resource sharing and isolation, such as VPCs, VPN tunnelling, and cross-project Virtual Private Cloud network peering.


3. What is the primary advantage of using a shared private cloud for collaboration between organizations?


Answer: Shared private clouds allow two or more organizations to collaborate on resources and projects while maintaining control over communication and access.


Application Questions


1. Design a network topology that balances resource sharing and isolation for a multi-organization project.


Answer: Consider using VPCs, VPN tunnelling, and cross-project Virtual Private Cloud network peering to achieve the desired balance of collaboration and security.


2. How can cloud architects ensure secure communication between components in a microservice architecture?


Answer: Implement identity access controls and encryption policies to control what kind of communication can occur between services.


3. What are some potential drawbacks of using cross-project Virtual Private Cloud network peering for collaboration between organizations?


Answer: Potential drawbacks may include increased complexity, security risks if not properly configured, and limitations in scalability.
