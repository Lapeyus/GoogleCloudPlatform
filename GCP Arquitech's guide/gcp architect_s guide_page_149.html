Unlike other cloud providers, GCP prices the compute and storage separately therefore you will need to consider the following three billable items when calculating the overall cost.

1. The type of Cloud instance, and the number of nodes in the instance.
2. The total amount of storage you use.
3. The amount of network bandwidth used by egress traffic.

The problem here is that although it is good that you are billed only for active instances and you pay only for the storage used, it is not easy to forecast if you have a large datasets. Moreover, the compute cost becomes less important as it is the same no matter if you choose an SSD or an HDD storage type.

For example, the difference in performance between SSD and HDD are not comparable. Although the timing to perform writes is the same for both cases, the timing for reads is about 20 times faster with SSD. Of course, there is a case for going with HDD as scans for HDD lag behind SSD by just 20%. Hence, if you know your access pattern is mostly going to be scans, then HDD option might be a good option as HDD storage is only 15% of the cost of SSD.

BigQuery
BigQuery is Google's Cloud-based version of a data warehouse. Unlike Bigtable, it targets data as a big picture and can run a query against vast volumes of data in a short time. This is because in BigQuery the data is stored in columnar data format so it is much faster in scanning large amounts of data compared with Bigtable.

In addition, BigQuery allows you to scale to petabyte and is targeted at the enterprise data warehouse for analytics market space. BigQuery is serverless, which means it is a fully managed service. Serverless computing also benefits from seamless scalability as computing resource can be spun up on-demand without involving administrators managing any infrastructure.

As a result of this transparent scalability, BigQuery can scan Terabytes of data in seconds and Petabytes of data in minutes. On the other hand with regards data ingestion, BigQuery integrates to load or stream data from/to Google Cloud Storage, or Google Cloud Datastore.

However, we must remember that BigQuery is designed as a data warehouse so it is best for analytical (OLAP) type of query and scanning large amount of data and is not designed for transaction type queries (OLTP). For example, for small read/writes, it takes about 2 seconds while Bigtable takes about 9 milliseconds for the same amount of data. Bigtable is much better off for OLTP type of queries.

Although BigQuery supports atomic single-row operations, it lacks cross-row transaction support.

For pricing, there are two major components in the cost of using BigQuery: Storage Cost and Query Cost.

Understanding GCP Pricing and BigQuery


GCP prices compute and storage separately, making it essential to consider three billable items when calculating overall costs:


    

  • Cloud instance type and number of nodes: The type of Cloud instance and the number of nodes in the instance impact costs.
    • 

  • Total storage usage: Storage costs are billed based on actual usage.
    • 

  • Network bandwidth used by egress traffic: Egress traffic is charged separately from compute instances.
    • 



Challenges with Forecasting Costs


Forecasting costs for large datasets can be challenging due to the lack of comparable performance between SSD and HDD storage types. While SSDs offer faster reads, HDDs are more suitable for scans, which may impact overall cost decisions.


BigQuery Overview


BigQuery is a cloud-based data warehouse that targets data as a big picture. It offers:


    

  • Columnar data format: Faster scanning of large amounts of data compared to Bigtable.

  • Serverless computing: Fully managed service with seamless scalability, allowing for on-demand resource allocation without administrative involvement.

  • Petabyte-scale storage: Supports large datasets and is targeted at the enterprise analytics market space.

BigQuery is designed for analytical (OLAP) type queries and scanning large amounts of data. It is not suitable for transactional (OLTP) type queries, which require faster read/write operations. BigQuery supports atomic single-row operations but lacks cross-row transaction support.


Pricing Components

Compute and Storage Costs

Topic 2


BigQuery Overview

Topic 3


BigQuery Limitations and Pricing
BigQuery's limitations include its design for analytical (OLAP) queries rather than transactional (OLTP) queries. It also lacks cross-row transaction support. The cost of using BigQuery consists of two major components: Storage Cost and Query Cost, with pricing varying based on these factors.


Topic 4


Comparison to Bigtable
Bigtable is compared to BigQuery in terms of performance for small read/writes, where Bigtable outperforms BigQuery. However, BigQuery excels in scanning large amounts of data due to its columnar data format and seamless scalability.


Topic 5


Storage Type Impact on Performance
The choice between SSD and HDD storage has a significant impact on performance, with SSDs offering faster reads but slower writes compared to HDDs. The optimal storage type depends on the access pattern, with HDDs being suitable for scans and SSDs being better suited for general-purpose use.


Topic 6


Seamless Scalability
Serverless computing benefits from seamless scalability, allowing resources to be spun up on-demand without requiring administrator intervention. This transparent scalability enables BigQuery to efficiently handle large-scale analytics workloads.
