So let's start with taking a look at the capacity planning cycle. First, you will have to start with a forecast. This will likely be based on the project plan, marketing data or previous industry or product knowledge. Then you have to decide what resources are going to be necessary, and then try to go and get approval for that. But since cloud deployments are much more influenced by operation expense (OpEx) than upfront capital expense (CapEx), this is likely to be forthcoming and then you can go ahead and deploy.

The next stage is the monitor and measure. This would have started during testing and commissioning but it is only once you move into production that it starts to have real value. This is because you can now monitor the indicators in production. Now you will know the real world conditions rather than on pre-production forecasts so you can adjust and predict future trends and optimise accordingly.

Then simply repeat that process and each time your forecasts should become more accurate. So when it comes to these iterative forecasts, the purpose is to monitor and predict future demand, and plan for future growth. Therefore once in production we compare the actual measured conditions against the forecasts, to see whether they were high or low, and we will try to make a more accurate prediction next time.

So this way, forecasting converges on a practical value by learning from previous errors. Then consider what other values should you potentially include in your forecasting estimate and give yourself some over provisioning to cater for unforeseen happenings - this is called an error budget.

Forecasting estimation and measuring simply beats using advanced analytics and machine learning have taken a lot of the guess work out of predicting trends, mitigating the risk of provisioning. Nonetheless, as we said earlier, cloud deployments are about managing on-going operational expense rather than the upfront capital expense so overprovisioning is not such a large risk.

Indeed, we actually need to overprovision for instance when building resiliency so we use a n+2 model to cope with aggregated traffic in the case of a node failure. There is also a requirement for meeting non-linear demand such as peak hour traffic or volatile spikes in system load.

So the dilemma is when dealing with instance overhead estimation, how do you overestimate without overestimating too much? Well, we know we have to provide some overhead for such things as spikes in demand and the like. There is also the requirements of fail-over in the case where a VM is acting in a redundancy configuration such as in a managed instance.

So we must take all these known production overheads into consideration. However, we must also not forget the inherent capacity overheads within a VM, such as in running the OS. This means that if the VM is already consuming 30% of its CPU capacity just to run itself, then you're only going to get another 30%, to run the application.

This is because you will need the remaining 40% to cater for fail-over in a managed instance design where it will have to accept the load of a failed instance. We allocate a generous 30% of capacity for the partial load for this burden and any unknown overheads that may occur in production. This leaves us with 10% spare capacity because it is not a good idea to plan to run at 100% due to the very real risk of introducing potential cascading failures.

Capacity Planning Cycle Overview


The capacity planning cycle involves several stages, starting with forecasting based on project plans, marketing data, or industry/product knowledge. This stage is followed by obtaining approval for necessary resources and deploying them.


Iterative Forecasting Process

Comprehension Questions


    

  1. 

    What are the three main stages of the capacity planning cycle?
Answer: The three main stages of the capacity planning cycle are forecast, monitor and measure, and repeat.
    

    2. 

  2. 

    Why is it important to include an error budget in forecasting estimates?
Answer: An error budget allows for over-provisioning to cater for unforeseen happenings, ensuring that there is sufficient capacity to handle unexpected spikes in demand or system load.
    

    3. 

  3. 

    What is the purpose of using a n+2 model in cloud deployments?
Answer: The n+2 model is used to cope with aggregated traffic in the case of a node failure, providing a buffer to ensure that the system can continue to operate even if one instance fails.
    

    4. 



Analytical Questions


    

  1. 

    How does the nature of cloud deployments influence capacity planning compared to traditional on-premises deployments?
Answer: Cloud deployments are more influenced by operational expenses (OpEx) than upfront capital expenses (CapEx), which means that over-provisioning is not as large a risk, but still necessary for resiliency and meeting non-linear demand.
    

    2. 

  2. 

    What are some of the key factors to consider when estimating instance overheads?
Answer: Factors include spikes in demand, fail-over requirements, inherent capacity overheads within a VM (e.g., running the OS), and the need for spare capacity to avoid cascading failures.
    

    3. 

  3. 

    How does the concept of an error budget relate to the use of advanced analytics and machine learning in forecasting?
Answer: Advanced analytics and machine learning can help mitigate the risk of provisioning by providing more accurate forecasts, but it is still important to include an error budget to account for unforeseen events.
    

    4. 



Application Questions


    

  1. 

    Suppose you are planning a cloud deployment for a new e-commerce application that is expected to experience peak hour traffic during the holiday season. How would you incorporate this information into your capacity planning?
Answer: You would need to estimate the increased demand and allocate additional resources (e.g., instances, storage) to handle the expected spike in traffic.
    

    2. 

  2. 

    If you are using a n+2 model to cope with aggregated traffic in the case of a node failure, what is the optimal value for 'n'?
Answer: The optimal value for 'n' would depend on various factors, including the expected frequency and severity of failures, as well as the available resources and budget.
    

    3. 

  3. 

    How would you adjust your error budget if you were to use a cloud provider that offers auto-scaling features?
Answer: You may be able to reduce your error budget since the cloud provider can automatically add or remove instances based on demand, reducing the need for manual over-provisioning. However, it's still important to have some buffer in place to account for unexpected events.
