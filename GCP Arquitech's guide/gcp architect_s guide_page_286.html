Stackdriver Logging


Overview


You should be aware that log entries exported to Cloud Storage are batched and sent out approximately every hour. On the other hand, log entries exported to BigQuery and Cloud Pub/Sub are streamed to those destinations immediately.


When exporting logs to different storage classes of cloud storage, you should consider how long and often you will need to access the data. This is because Stackdriver logging will only retain the logs for 30 days.

Error Reporting


Stackdriver has another in-built feature that is extremely useful - error reporting. Indeed Stackdriver error reporting will count, analyse and aggregate all of the errors in all of your active cloud resources.


This works because the Stackdriver probe or agent sends data to a centralized error management interface. The centralized error reporting node then collates and displays the results with granular sorting and filtering capabilities.


Error reporting as a function is available for the App Engine Standard environment, the App Engine Flexible environment, and the Compute Engine services. Also, it is available in terms of programming languages, using Java, Python, JavaScript, Ruby, C#, PHP, and Go as the exceptions stack trace parser can handle these languages.


Tracing


Stackdriver Trace is a distributed tracing system that collects latency data from your applications and displays it in the GCP console. Using Stackdriver trace you can track how requests propagate through your application and receive detailed new real-time performance insights.

Stackdriver Logging best practices
The best practices when it comes to Stackdriver logging are:
• Don't use substring matches on service names or resource types. For example, searching for the text 'abc' will match log entries containing 'abc', 'xyabcyx', and even 'ABC'. Instead, for an exact match, use the equals operator.
Perform faster searches by targeting specific values of indexed fields. For example, specify your search for a log entry's name, resource type, or resource label. Using advanced filters will result in even more effective queries. You can add filters directly from log entries.
• Finally, get accustomed to using and learning the advanced viewing interface. Start by setting up selections and filters in the basic interface, and then switch to the advanced interface.

Stackdriver has another in-built feature that is extremely useful - error reporting. This works because the Stackdriver probe or agent collects data from a distributed collection point that sends it to a centralized error management interface. The centralized error reporting node then collates and displays the results. Furthermore, it does so with granular sorting and filtering capabilities, so that you can even configure and subsequently trigger notifications and alerts in real-time when predetermined anomalies are detected.

Error reporting is available for the App Engine Standard environment, the App Engine Flexible environment, and the Compute Engine services. Also, it is available in terms of programming languages, using Java, Python, JavaScript, Ruby, C#, PHP, and Go as the exceptions stack trace parser can handle these languages.

Analytical Questions


    

  1. 

    Why is it recommended not to use substring matches on service names or resource types when searching for log entries?
Answer: Using substring matches can lead to false positives, as the search will match log entries containing the specified text even if it's not an exact match.
    

    2. 

  2. 

    How does targeting specific values of indexed fields in a search query improve performance?
Answer: Targeting specific values of indexed fields allows for more precise searches and reduces the number of irrelevant results, resulting in faster search times.
    

    3. 

  3. 

    What is the benefit of using advanced filters in log entry searches?
Answer: Advanced filters enable more effective queries by allowing users to specify exact matches or narrow down search results based on specific criteria.
