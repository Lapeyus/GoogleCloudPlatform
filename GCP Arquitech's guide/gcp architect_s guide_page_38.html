Basic Cloud Networking
Google's GCP provides you with the concept of the VPC which to all 
intent and purposes resembles your own private network. It also 
provides the underlying networking required to connect your cloud 
resources to each other and the internet. This networking service is 
typically automated and requires no intervention from you.


However if you like you can do many of the network tasks that you perform 
on your on-premises network such as you can segment your networks, use 
firewall rules to restrict access to instances, and create static routes 
to forward traffic to specific destinations. However there is a key 
difference between your on-premise networking and Google GCP networks.

Google's GCP provides you with the concept of the VPC which to all intent and purposes resembles your own private network. It also provides the underlying networking required to connect your cloud resources to each other and the internet. This networking service is typically automated and requires no intervention from you.

However, if you like you can do many of the network tasks that you perform on your on-premises network such as segmenting your networks, using firewall rules to restrict access to instances, and creating static routes to forward traffic to specific destinations. There is a key difference between your on-premise networking and Google GCP networks.

Google's GCP is built upon a Software Defined Network which can be considered to be one vast layer-2 network with global reach. Thus the Virtual Private Cloud networks that you define will have global scope. This means that they can have subnets in any GCP region worldwide and importantly, the subnets can span the zones that make up a region.

This architecture makes it easy for you to build redundancy into your design as there will be very low latency between resources in different zones as they are on the same subnet and communicating at layer-2 much like they were connected to the same switch. This is perfect for business continuity as a failure in one zone can be addressed seamlessly by the resources in another zone.

Moreover, the SDN architecture makes it easy to define your own network layout with a global scope. In addition to supporting resources in different zones on the same subnet, you can dynamically increase the size of a subnet in a custom network by expanding the range of IP addresses allocated to it without affecting the already configured VMs.

Google Cloud Platform Virtual Private Cloud (VPC)


Overview


A Virtual Private Cloud (VPC) is a critical component of Google Cloud Platform (GCP), providing global and flexible networking capabilities to connect cloud-based resources and services. It resembles a private network, offering automated networking with the ability to segment networks, use firewall rules, and create static routes.


Key Features


    

  • Global Scope: VPCs have a global scope, allowing subnets in any GCP region worldwide.
    • 

  • Zone Spanning: Subnets can span zones within a region, enabling low latency between resources in different zones.
    • 

  • Business Continuity: Redundancy is built into the design, ensuring seamless addressing of failures in one zone by resources in another zone.
    • 

  • SDN Architecture: The Software Defined Network (SDN) architecture makes it easy to define network layouts with global scope.
    • 



Benefits


    

  • Easy Network Design: Define your own network layout with a global scope, supporting resources in different zones on the same subnet.
    • 

  • Dynamic Subnet Size: Increase the size of a subnet by expanding IP address ranges without affecting already configured VMs.
    • 

  • Automated Networking: Typically automated, requiring no intervention from you.
    • 



Creating a VPC


You can create a VPC inside your first GCP project or choose the default VPC. Regardless, your VPC provides networking capabilities and services to connect Google Cloud platform resources to each other and to the internet.


Regions and Zones


VPCs are available in various regions, including US-West and US-East. Each region is composed of multiple zones, which can be spanned by subnets within a VPC.

Comprehension Questions


1. What is the primary purpose of creating a Virtual Private Cloud (VPC) in Google Cloud Platform (GCP)?


Answer: To provide critical internetworking between resources such as VMs in a global and flexible way.


2. How can you create a VPC in GCP?


Answer: You can either create it inside your first GCP project or choose the default VPC and use that.


3. What is the key difference between on-premise networking and Google GCP networks?


Answer: Google's GCP is built upon a Software Defined Network (SDN) which provides a global layer-2 network with low latency between resources in different zones.


Analytical Questions


1. How does the SDN architecture of GCP enable redundancy in VPC designs?


Answer: The SDN architecture allows for subnets to span across zones, enabling low latency and seamless communication between resources in different zones.


2. What are the benefits of using a custom network layout with a global scope in GCP?


Answer: It provides flexibility to define your own network layout, dynamically increase subnet size without affecting already configured VMs, and supports resources in different zones on the same subnet.


3. How does the VPC architecture address business continuity in case of a failure in one zone?


Answer: The low latency between resources in different zones due to the SDN architecture enables seamless communication and addressing of failures in one zone by resources in another zone.


Application Questions


1. Design a VPC for a web-server application that requires high availability and redundancy.


Answer: Create a custom network layout with multiple subnets spanning across zones, use firewall rules to restrict access, and dynamically increase subnet size as needed.


2. How would you implement business continuity in a GCP-based application using VPCs?


Answer: Use the SDN architecture to create redundant connections between resources in different zones, ensure low latency, and define custom network layouts with global scope.


3. What are some best practices for managing VPCs in GCP?


Answer: Regularly review and update firewall rules, monitor subnet size and adjust as needed, and use the default VPC or create a custom VPC based on application requirements.
