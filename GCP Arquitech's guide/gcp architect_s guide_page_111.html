Compute Engine Roles


Overview


Compute Engine roles provide access to various features and functionalities within the platform.


Roles


    

  • roles/compute.networkUser: Provides access to a shared VPC network once granted, service owners can use VPC networks and subnets that belong to the host project.
    • 

  • roles/compute.orgSecurityPolicyAdmin: Full control of Compute Engine Organization Security Policies.
    • 

  • roles/compute.storageAdmin: Permissions to create, modify, and delete disks, images, and snapshots.
    • 



Combining Roles


Roles can be combined to give users or groups specific permissions. For example, granting roles/compute.networkAdmin and roles/compute.securityAdmin allows a networking team to manage networks and apply firewall rules.

Overview


Understanding the lifecycle of a VM is crucial for knowing its status at any given time, especially for billing purposes.


Stages


    

  • Provisioning State: The initial stage after configuring VM properties. Resources are reserved but the VM itself isn't running.

  • Staging State: The next stage where resources are moved from being reserved to acquired and the instance is prepared for launch.

  • Running State: Where the VM starts doing useful work, executing pre-configured startup scripts, and enabling SSH or RDP access.

Benefits


The running state offers flexibility with tasks such as:
- Moving VMs to different zones
- Taking snapshots of persistent disks for backup or migration purposes
- Exporting system images
- Re-configuring metadata


However, some admin tasks require stopping the virtual machine.

Roles in Compute Engine
These are just some of the predefined roles within Compute Engine that serve specific job functions and roles. However, you can combine several roles and give them to a user or group. For example, if your networking team is responsible for managing the network and applying firewall rules, you can grant roles/compute.networkAdmin and roles/compute.securityAdmin to allow the networking team to do only those specific tasks.

VM Lifecycle
It is essential to understand the lifecycle of a VM so that you know the exact status of the VM at any given time - for billing purposes. The lifecycle of a VM is represented by different stages and their related status.

The initial stage is the provisioning state, which occurs after configuring all properties of a VM such as CPU, memory, disk, OS, and clicking create. Once the VM goes through the internal build process, it enters the provisioning state. At this point, all the VM's resources such as CPU, memory, and disks are being reserved, but the VM itself isn't running at this stage.

The next stage is when the newly provisioned VM moves to the staging state. This is where the resources have been moved from being reserved to acquired, and the instance is prepared for launch. It is in this staging state that the Compute Engine is configuring the resources and adding IP addresses as well as starting the booting up process for the OS image.

Once the staging stage completes and the system has booted, the VM is now in the running state. This is the state where it will start doing useful work such as executing pre-configured startup scripts and enabling SSH or RDP access. The running state is where the VM is fully operational and can do all the tasks that any physical server could manage.

However, a VM can do a lot of things that you can't achieve on a physical server. For example, you can live-migrate your virtual machine from one host to another host in the same zone instead of requiring an instance to be rebooted.

This feature is not only beneficial to you as it also allows Google to perform maintenance tasks critical to keeping the infrastructure running. Indeed, the live migration feature may work transparently to you as Google works behind the scenes to keep the infrastructure protected and reliable without any interruption to your service on any of your VMs.

The great thing from an administration perspective is that there are many tasks you can complete while the VM is running. These include the ability to move your VM to a different zone, take a snapshot of the VM's persistent disk for backup or for VM migration purposes, exporting the system image, or re-configuring metadata. However, not all admin tasks can be performed on the fly as some actions require you to stop your virtual machine. Some of the tasks that will require you to enter the stopped state are if you want to upgrade your machine by adding more CPUs.
