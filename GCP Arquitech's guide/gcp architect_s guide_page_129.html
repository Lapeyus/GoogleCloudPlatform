User-Managed Keys


Overview


These keys cannot be downloaded but Google will keep the keys.


User-managed keys are created, downloadable, and managed by users. When you create a new key pair, you download the private key, which is not retained by Google. With external keys, you are responsible for key privacy and secure storage of the private key and other management operational chores, such as key rotation.


Key Rotation


Each service account can support one or more keys and these are used to authenticate with Google Cloud APIs. The specific keys should be periodically rotated, which involves a process where you will generate a new version of the key. This is because the fact that you may well be distributing the key amongst users will make it susceptible to leakage so it is better to rotate the keys regularly and remove the old version of a key.

User-managed Keys and External Keys


User-managed keys are created, downloadable, and managed by users. When creating a new key pair, users download the private key, which is not retained by Google. In contrast, external keys are managed by Google and should be periodically rotated to prevent leakage.


External keys are used to authenticate with Google Cloud APIs, and each service account can support one or more keys. Regular rotation of these keys limits the risk of them being used by malicious third-party users in the event of key compromise.


IAM Best Practices


When designing your own IAM policies, it is essential to have a good understanding of the GCP IAM resource hierarchy. This allows you to leverage concepts like inheritance and isolation between various tiers.


Establishing projects helps group and share resources with the same trust level. When granting roles, recognize the effects of policy inheritance and use the principle of least privilege to minimize permission creep.


Regularly audit the memberships of groups used in policies using Cloud audit logs. Granting roles to groups instead of individuals eases administration and reduces errors, except for high-risk tasks in high security environments where individual assignments are preferred.


Sub-divide specific roles into multiple groups to assign privileges more granularly for finer control. Update group membership rather than changing a Cloud IAM policy whenever possible.
