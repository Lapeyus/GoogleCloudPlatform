Cloud Resource Management


Overview


Cloud resource management involves organizing resources into a hierarchical structure, with each level having its own set of roles and permissions.


# Roles and Permissions


## Folder Roles


    

  • Admin: Full administrative control over all folders.
    • 

  • Creator: Browse folder hierarchy, create folders.
    • 

  • Viewer: View folders and projects, view sub-folders.
    • 



## Project Roles


    

  • Owner: Create new products, automatically owner.
    • 

  • Project Leader: Deletion privileges for projects.
    • 

  • Viewer: View projects.
    • 

  • Editor: Edit projects.
    • 



# Product-Specific Roles


    

  • Compute Engine's Instance Admin role: Create, modify, delete virtual machine instances and disks.
    • 



## Cloud IAM Policy


Resource Manager policies secure projects and resources. However, strict control over policy changes and group memberships is necessary to prevent permissions creep.


Benefits of Hierarchical Structure


    

  • Autonomy for teams and departments.
    • 

  • Delegation of administration rights at the folder level.
    • 

  • Restricted access to resources in particular folders.
    • 



# Example Use Case


A company has a hierarchical structure with organizational, folder, project, and resource levels. Each level has its own set of roles and permissions, ensuring that users have the minimum necessary access to perform their tasks.

  • Apply the minimum access required to perform a role.

# Conclusion


Cloud resource management involves organizing resources into a hierarchical structure, with each level having its own set of roles and permissions. A well-designed hierarchy ensures that users have the minimum necessary access to perform their tasks, while preventing permissions creep.

Each team folder could contain several additional subfolders to represent different applications, development areas, test-beds, or staging areas. Folders also allow autonomy as they allow delegation of administration rights at the folder level. So, for example, each head of a department can be granted full viewer and auditing roles for all GCP resources that belong to their departments.

Similarly, access privileges can also be restricted to prevent access to resources in particular folders. Users in one department only gain access and create GCP resources within their own folder.

Resource Manager Roles can be applied while keeping in mind the best practice of applying the minimum access required to perform a role and that policies are inherited top to bottom. Indeed, we have seen this applied in the top-level organisational node. However, the folder node also has multiple roles that mimic the organisational roles, but are applied only to resources within a folder.

There is an admin role that provides full administrative control over all folders. There is also a creator role, which enables browsing of the folder hierarchy and creating new folders. Additionally, there is a viewer role, which allows members to view folders and their projects and subfolders.

Similarly, when we look at roles at the project level, we see that there is a creator role that allows a user to create new products, making that user automatically the owner. There's also a project leader role that grants deletion privileges for projects.

Project Roles Overall, we have seen that at each level of the hierarchy such as at the organization, folder, project, and resource - we have specific roles. Project roles mimic the primitive roles in that there is a viewer, editor, and owner role, and all these roles are concentric. A project owner can also invite others to become project owners.

Extending an invitation is done by sending an email from the GCP console and having the prospective project owner accept the invitation. This type of email invitation is only available for project owner roles and not for other roles.

With regards to the many product-specific roles, like the Compute Engine's Instance Admin role, which provides permissions to create, modify, and delete virtual machine instances and disks. These are resource and product specific so each product or service has several of these roles.

Cloud IAM Policy Resource Manager policies can secure your projects and their resources but you also want more granular control as to how users will gain access to resources through the application of IAM policies and group memberships. Without strict control over policy changes and group memberships, you tend towards permissions creep.

This is what tends to happen in all environments and organisations when the users, typical due to changes in role, eventually and inadvertently gain more permissions than they need, which violates the principle of least privilege.

Organizational Structure and Access Control in GCP


Each team folder can contain multiple subfolders representing different applications, development areas, test-beds, or staging areas. This allows for autonomy at the folder level, enabling delegation of administration rights to department heads, who can be granted full viewer and auditing roles for all GCP resources within their departments.


Resource Manager Roles


Resource Manager Roles provide a hierarchical structure with multiple roles that mimic organizational roles but are applied only to resources within a folder. These roles include:


    

  • Admin role: provides full administrative control over all folders
    • 

  • Creator role: enables browsing the folder hierarchy and creating new folders
    • 

  • Viewer role: allows members to view folders, projects, and subfolders
    • 



Project Roles


Project Roles follow a concentric structure with viewer, editor, and owner roles. A project owner can invite others to become project owners by sending an email invitation through the GCP console.


Product-Specific Roles


Product-specific roles, such as Compute Engine's Instance Admin role, provide permissions for specific resources and services. Each product or service has multiple roles that cater to different needs.


Cloud IAM Policy


Resource Manager policies can secure projects and resources but may not provide enough granular control over user access. Applying strict controls over policy changes and group memberships is crucial to prevent permissions creep, which violates the principle of least privilege.
